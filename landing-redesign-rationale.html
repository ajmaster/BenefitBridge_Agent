Before and after critique

The current marketing surface is cleaner than the original, but it still reads like a split hero plus repeated cards. The next pass should let the product behavior carry the explanation.

Current risk

The page can still feel explanatory: headline, paragraph, card grid, demo gallery, and trust strip as separate blocks.

  • Hero media is useful, but it is treated like a side object.
  • Workflow steps are individual cards, not one connected system.
  • Boundary copy is correct but can feel detached from the product story.
  • Demo assets need to stay visually aligned with the current UI.

Target improvement

Use a product-led story: the agent conversation is the center, and every section shows how the conversation becomes action.

  • Hero becomes a live-feeling workflow scene.
  • Scroll sections reveal connected states instead of isolated cards.
  • Safety and source boundaries appear inside the workflow.
  • Remotion demo reinforces the same chat-to-packet loop.

Visual direction: Product Atlas

Product Atlas is the recommended default. It borrows Inngest's terse, product-led, connected-motion structure while preserving AidAtlasCA's civic trust and safety constraints.

Borrow from Inngest

  • Short first-viewport copy.
  • Product proof as the main visual.
  • Connected workflow primitives.
  • Before/after transformation.
  • Scroll-staged line drawing and card lift.

Translate for AidAtlasCA

  • Conversation replaces code as the hero object.
  • Source-backed paths replace execution primitives.
  • County-level Bay Area support replaces infrastructure scale.
  • Prep packet assembly replaces build/deploy outcomes.
  • Safety boundaries replace enterprise claims.

Maps guidance

Use Google Maps Embed only as an optional visual enhancement. Build iframe queries from curated organization plus city/county, not user exact addresses. If no public embed key is configured, show the existing safe fallback and outbound Google Maps links.

Safety and copy boundaries

These statements should be visible in the product story, not hidden only in footer legal copy.

Eligibility Official agencies decide eligibility and amounts.
Applications AidAtlasCA prepares packets but does not submit applications.
Location Use city/county/ZIP only, not exact addresses.
Local handoffs Local details can change. Call before going to confirm current availability.

Concept prompts

If visual concepts are generated before coding, use these three directions. Product Atlas is the recommended default.

Product Atlas

The strongest default: a product-led hero with chat as the main object, connected workflow rails, and crisp source/packet modules. It should feel closest to the Inngest inspiration while remaining civic and safe.

Civic Workflow

A warmer accessibility-forward interpretation with less technical energy and more public-service calm. Best if trust needs to outrank visual edge.

Bay Area Signal

A regional map-forward concept that makes nine-county coverage more prominent. Best if resource navigation becomes the primary marketing proof.

Acceptance checklist

The design package is ready for implementation only when these checks are met.

Final result target: the landing page explains AidAtlasCA through a visible chat-to-packet workflow, keeps boundary copy visible, supports English and Spanish, treats maps as optional and safe, separates Remotion from frontend runtime, and contains no forbidden eligibility, amount, application, upload, live-availability, route, near-me, or exact-address claims.
